# 메인 검색 기능 구현 - 단계별 가이드

> **목표**: 메인 페이지에서 도시, 체크인/체크아웃, 객실 수로 검색 → 조건에 맞는 Room 목록 조회 (QueryDSL)

---

## 전체 순서 요약

| 단계 | 하는 일 | 위치 |
|------|---------|------|
| **1** | QueryDSL 의존성 추가 | `build.gradle` |
| **2** | Q클래스 생성 설정 | `build.gradle` + 빌드 실행 |
| **3** | 검색 요청/응답 DTO 정의 | `domain/hotel/dto/` |
| **4** | QueryDSL 검색 쿼리 작성 | Repository (Custom + Impl) |
| **5** | 서비스에서 검색 메서드 추가 | `HotelService` |
| **6** | 컨트롤러에 검색 API 추가 | `MainPageController` 또는 `HotelController` |
| **7** | 메인 페이지 폼 → 검색 요청 연결 | `index.html` |
| **8** | 검색 결과 페이지 (뷰) | `templates/` + 컨트롤러 뷰 반환 |

---

## Step 1: QueryDSL 의존성 추가

**파일**: `build.gradle`

**할 일**:
- `querydsl-jpa` (JPA용 QueryDSL)
- `querydsl-apt` (Q클래스 생성용, `annotationProcessor`)

**이유**: QueryDSL로 동적 쿼리(도시 + 기간 + 재고)를 짜려면 라이브러리와 Q클래스가 필요합니다.

---

## Step 2: Q클래스 생성

**할 일**:
1. `build.gradle`에 Q클래스 출력 경로 설정 (옵션, 기본은 `build/generated/...`).
2. 터미널에서 `./gradlew compileJava` 실행 → `Hotel`, `Room`, `RoomInventory`, `Address` 등에 대한 Q클래스 생성.

**확인**: `build/generated/sources/annotationProcessor/.../QHotel.java` 등이 생성되면 성공.

---

## Step 3: 검색용 DTO 정의

**위치**: `src/main/java/com/travelGo/domain/hotel/dto/`

**만들 파일**:
- **검색 요청**: `HotelSearchRequest.java`  
  - `city` (String), `checkIn` (LocalDate), `checkOut` (LocalDate), `roomCount` (int)
- **검색 결과(한 건)**: `RoomSearchResult.java` (또는 `HotelRoomSearchResult`)  
  - 화면에 보여줄 필드: 호텔명, 객실명, 가격, 기간 등

**이유**: 컨트롤러는 DTO로 받고, 서비스/Repository는 이 조건으로 QueryDSL 쿼리를 실행합니다.

---

## Step 4: QueryDSL 검색 쿼리 (Repository)

**위치**: `domain/hotel/repository/`

**구조**:
- `RoomSearchRepository.java` (인터페이스) — 검색 메서드 시그니처.
- `RoomSearchRepositoryImpl.java` (구현체) — QueryDSL로 실제 쿼리.

**쿼리 로직 요약**:
1. `RoomInventory` 기준으로 `Room` → `Hotel` 조인.
2. `Hotel.address.city` = 검색 도시 (동등 조건).
3. `RoomInventory.date` ∈ [체크인, 체크아웃), `availableCount >= roomCount`.
4. Room 기준 `GROUP BY room.id`.
5. `HAVING count(*) = (체크아웃 - 체크인) 일수` → 기간 내 매일 재고 있는 Room만 선택.
6. 이 조건을 만족하는 `Room`(또는 Room ID) 목록 반환.

**이유**: “해당 기간 동안 매일 재고가 충분한 Room”만 찾기 위해 QueryDSL로 조인·그룹·having을 한 번에 처리합니다.

---

## Step 5: 서비스에서 검색 메서드

**파일**: `HotelService.java`

**할 일**:
- `searchAvailableRooms(HotelSearchRequest request)` 같은 메서드 추가.
- 내부에서 `RoomSearchRepository.searchAvailableRooms(request)` 호출.
- 반환 타입: `List<RoomSearchResult>` 또는 `List<Room>` 등 (정한 DTO에 맞춤).

**이유**: 컨트롤러는 서비스만 부르고, 비즈니스 로직·트랜잭션은 서비스에 두기 위함.

---

## Step 6: 컨트롤러에 검색 API

**파일**: `MainPageController.java` 또는 `HotelController.java`

**할 일**:
- GET (또는 POST) 하나 추가.  
  - 예: `GET /search?city=제주&checkIn=2025-02-10&checkOut=2025-02-12&roomCount=1`
- 파라미터를 `HotelSearchRequest`로 바인딩.
- `HotelService.searchAvailableRooms(...)` 호출.
- 검색 결과를 Model에 넣고 **검색 결과 전용 뷰**로 이동 (예: `searchResult.html`).

**이유**: 메인 페이지 폼이 이 URL로 요청을 보내고, 결과 페이지에서 리스트를 보여주기 위함.

---

## Step 7: 메인 페이지 폼 연결

**파일**: `templates/index.html`

**할 일**:
- 검색 영역을 `<form>`으로 감싸기.
- `action`: Step 6에서 만든 URL (예: `GET /search`).
- 입력 필드에 `name` 부여: `city`, `checkIn`, `checkOut`, `roomCount` (컨트롤러 파라미터명과 맞추기).
- “검색” 버튼을 `type="submit"`으로.

**이유**: 사용자가 입력한 값이 쿼리스트링(또는 POST body)으로 넘어가 검색이 실행되도록 하기 위함.

---

## Step 8: 검색 결과 뷰

**파일**: `templates/hotels/searchResult.html` (또는 `search.html`) 등.

**할 일**:
- 컨트롤러가 Model에 넣은 `rooms`(또는 `searchResults`) 리스트를 Thymeleaf로 반복 출력.
- 호텔명, 객실명, 가격, 체크인/체크아웃 등 표시.
- 각 항목에서 “예약하기” 등으로 상세/예약 페이지로 링크.

**이유**: 검색 결과를 사용자에게 보여주기 위함.
